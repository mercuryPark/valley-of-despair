# 2026-05-04 — 제품 피벗 설계 (cross-domain 큐레이션 학습 노트)

## 배경

기존 valleyofdespair는 "FE 1~5년차 대상, 본인 실무 경험 40% 비중으로 4개 FE 도메인 + 4개 보조 도메인을 깊이 학습"하는 컨셉이었다. 1주차 인프라와 2주차 콘텐츠 파이프라인 인프라까지 완료된 시점에 본인이 사용 의도와 차별화 방향을 재정의했다.

새 의도:

- 타겟을 2~5년차 개발자(직군 불문)로 확장
- 본인이 1차 사용자 (자가 학습 + 노트 큐레이션). 콘텐츠 품질 합격선 도달 시 커뮤니티 공개
- 본인 실무 경험 차별화 제거 → 외부 자료·LLM 보조·본인 큐레이션의 합성으로 전환
- 차별화 4 기둥: 분야 간 연결성 / 쉬운 설명 / 실무 시나리오 / 자가 점검

이 문서는 그 피벗을 박는다. 코드와 문서 모두 정리해서 이후 작업이 새 방향에 맞춰 굴러가게 한다.

## 제품 정체성

### 한 문장 정의

> **2~5년차 개발자가 분야별로 끊어져 있던 지식을 실무 기반 예시로 연결하며 쉽게 학습하고, 자가 점검으로 이해도를 확인할 수 있는 학습 노트형 플랫폼.**

### 타겟

- 주: 2~5년차 개발자 (FE/BE/풀스택/데브옵스 등 직군 불문)
- 1차 사용자: 본인 (자가 학습 + 노트 큐레이션)
- 2차 사용자: 커뮤니티 (콘텐츠 품질 합격선 도달 후 공개)
- 제외: 코딩 입문자, 1년차 미만

### 차별화 4 기둥 (모두 동등)

1. **분야 간 연결성** — 같은 개념이 다른 분야에서 어떻게 나타나는지 명시
2. **쉬운 설명** — 평이한 한국어, 짧은 단락, 다이어그램·예시 위주
3. **실무 시나리오** — 노드마다 "실무에서 부딪히는 케이스 + 풀이"
4. **자가 점검** — 체크리스트(개념) + 시나리오형 질문(응용)

### 콘텐츠 합성 비중

| 소스                                         | 역할                                       | 비중 |
| -------------------------------------------- | ------------------------------------------ | ---- |
| 외부 자료 (MDN·web.dev·공식 문서·기술블로그) | 1차 출처·정확성 보증                       | 50%  |
| LLM 보조                                     | 구조 초안·예시·자가 점검 질문 생성         | 40%  |
| 본인 큐레이션·검수                           | 정확성 점검·연결성 식별·실무 시나리오 발굴 | 10%  |

## 데이터 흐름·에러 처리·테스트

### 데이터 흐름

기존 그대로:

```
content/{domain}/{NN-slug}.mdx
  └→ gray-matter (frontmatter 파싱)
    └→ FrontmatterSchema.parse (Zod)
      └→ src/lib/content/loader.ts
        ├ listAllNodes
        ├ listPublishedNodes
        ├ listNodesByDomain(domain)
        ├ loadNode(domain, slug) → status:published만
        └ getAdjacentNodes(domain, order)
      └→ /learn/[domain] · /learn/[domain]/[slug] 페이지
```

### 에러 처리

기존 패턴 유지:

- 잘못된 도메인 → `notFound()` (404)
- 미존재 slug → `notFound()` (404)
- frontmatter 형식 오류 → 빌드 시 `content:check`에서 exit 1
- `prerequisites` 깨짐 → error (빌드 차단)
- `crossLinks` 깨짐 → warning (점진 작성 허용)

### 테스트 정책

- TDD 강요 X. 이번 피벗에 테스트 추가 작업 없음
- 검증은 `content:check` 스크립트로 충분
- E2E는 v1.5+ Playwright (출시 직전 스모크만)
- 자가 점검 섹션 검증(`## 자가 점검` 헤딩·`<details>` 존재 여부)은 MVP에 미포함. 콘텐츠 30개 즈음 패턴 굳어진 뒤 추가 검토.
