# 0005. 플랫폼: Next.js + PWA 단일, RN은 v2 이후

- **Status**: Accepted
- **Date**: 2026-05-03

## Context

초기 답변은 "웹 + 앱 둘 다". 검토 결과:

1. 학습 콘텐츠는 **긴 글 읽기 + 코드 블록 + 다이어그램** 중심 → 웹 친화적
2. MDX·코드 하이라이팅·검색 인덱싱 도구 생태계가 웹에 압도적
3. 1인 개발에서 두 플랫폼 동시 유지보수는 콘텐츠 80~120개 작성과 양립 불가
4. "나중에 RN 추가"는 사이드 프로젝트 통계상 70%가 추가 안 함

## Decision

- **MVP**: Next.js 15 (App Router) + PWA 단일
- **v2 이후**: 사용자 피드백 보고 RN/Expo 추가 검토
- **사전 준비 안 함**: 모노레포·디자인 토큰 분리 등 RN 전제 작업 MVP에 끌어들이지 않음 (YAGNI)

## Consequences

### 긍정

- SEO 자연스러움 (학습 자료는 검색 유입 큼)
- SSG/ISR로 정적 빌드 → 빠른 LCP, Vercel 무료 티어 충분
- 모바일 사용자도 PWA로 80% 만족 (오프라인 캐싱·홈화면 추가)
- 콘텐츠 작성에 시간 집중 가능

### 부정

- 네이티브 기능(푸시 알림, 카메라) 사용 불가 — 학습 사이트는 불필요
- 모바일 앱 스토어 노출 없음 — 학습 도구는 검색 유입이 주

## 거부된 방향

### Next.js + RN/Expo 별도 레포 (MVP부터)
- 1인 개발 부담 2배
- API/디자인 토큰/인증 공유 셋업이 출시 전 시간 다 잡아먹음
- 콘텐츠 80~120개 작성 시간과 양립 불가

### Expo Router 웹 빌드 (단일 코드베이스)
- MDX·코드 하이라이팅 도구가 RN 환경에서 추가 작업 필요
- SEO·SSG가 Next.js 대비 약함
- 학습 사이트 핵심 가치(검색 유입·긴 글 읽기)에 부적합

### 사전 모노레포 셋업 ("나중에 RN 추가" 전제)
- 통계상 RN 추가가 안 일어날 가능성 큼
- 사전 분리 비용을 미리 지불하고 회수 못 함 (YAGNI 위반)
- v1.1+에서 RN 도입이 명확해질 때 그때 분리

## v2 RN 도입 트리거

- 출시 후 사용자 피드백에서 "앱 필요" 명확히 모임
- 학습 외 기능(노트 작성·푸시·딥링크) 필요성 발생
- 그 전에는 Next.js + PWA로 충분

## 면접 어필

"왜 RN 안 썼나"에 대한 답:
> RN/Expo 4년차 경험이 있지만, 학습 플랫폼의 핵심 가치(긴 글 읽기·SEO·검색 유입)에는 Next.js + PWA가 적합하다고 판단했다. 사용자 피드백에 따라 v2에서 RN 도입을 재평가할 것이다.

→ "잘 다루는 도구를 억지로 끼워넣지 않고 컨셉에 맞는 선택을 했다"는 어필.
