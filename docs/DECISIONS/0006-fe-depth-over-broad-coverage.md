# 0006. FE 깊이 우선, BE/DB는 보조 (얕게)

- **Status**: Accepted
- **Date**: 2026-05-03

## Context

작성자(Hoyeon, FE 4년차)는 "FE에 국한하지 않고 BE/CS도 학습하면서 만들고 싶다"는 동기를 가짐. 검토 결과 다음 우려 식별:

1. **콘텐츠 부담 폭증**: FE 4 도메인 60~80 노드도 큰 작업, 여기에 BE/DB 추가 시 150+ 노드
2. **본인 비전문 영역의 검수 불가**: LLM이 만든 BE 글의 미묘한 오류를 본인이 잡아낼 수 없으면 표면 검수만 됨
3. **roadmap.sh 정면 충돌**: "FE+BE+DB+CS 다양하게"는 roadmap.sh가 점유한 자리, 글로벌 스케일
4. **포트폴리오 어필 약화**: "FE 4년차가 비전문 영역까지 다룬다"는 깊이보다 분산을 보여줌
5. **"만들면서 학습"의 함정**: 학습 깊이 부족 + 개발 속도 폭락의 이중 실패 패턴

## Decision

- **메인**: FE 4 도메인 깊게 (각 15~20 노드, 총 60~80)
  - 렌더링 / 상태관리 / 성능 / 비동기·네트워크
  - 본인 깊이 + OfficeNEXT 실무 경험 → 차별화 핵심
- **보조**: FE 시점 인접 영역 얕게 (각 5~10 노드, 총 20~40)
  - HTTP·네트워크 (FE 시점) / 인증·인가 / DB 기초 / 서버 라이프사이클
  - "FE 개발자가 알아야 할 만큼"으로 한정, 깊이 욕심 X
- **합계**: 80~120 노드

## Consequences

### 긍정

- 1인 개발 가능 범위
- 본인 전문성에 정확히 fit
- 시장 갭(한국어 + FE 깊이)에 박힘
- 면접 어필 명확 ("FE 깊이 + 인접 영역 정리")
- "왜 FE만?"에 자연스러운 답 ("MVP 검증 후 확장")

### 부정

- "다양한 영역을 다룬다"는 매력 부족 → 보조 트리로 일부 해소
- 본인의 BE/CS 학습 욕구는 별도 채널로 (책·강의·실무)

## 운영 원칙

- **메인 트리 60~80 노드 완성 전에는 보조 안 건드림** (우선순위)
- 보조 트리 작성 시에도 깊이 욕심 X (얕고 정리 위주)
- UI에서 "메인 / 보조" 명확히 구분 (예: "FE 개발자가 알아야 할 BE 기초")

## 거부된 방향

### FE+BE+DB+CS 모두 깊게

- 1인 개발 비현실적
- 본인 비전문 영역 검수 불가
- roadmap.sh와 정면 충돌

### roadmap.sh 스타일 (넓고 얕게)

- 각 노드가 짧은 개관 + 외부 링크 위주
- roadmap.sh 한국 버전이 됨 → 차별화 약함

### 한 도메인만 (예: 렌더링만 깊게)

- "FE 핵심 도메인 다 다루지 못함"이 약점
- 4 도메인은 면접 빈도·실무 깊이·도메인 간 연결성 모두 자연스러움

## "본인도 학습하고 싶다"의 처리

이 동기는 분리해서 다룬다:

- **학습은 별도 채널로**: 책, 강의, 실무, 블로그
- **만드는 건 잘 아는 것 위주**: FE 깊이 + 본인 경험
- 보조 트리는 "_학습 결과를 정리_"하는 영역 (새로 학습이 아니라 정리)

이를 인정하지 않으면 "만들면서 학습" 함정에 빠져 둘 다 부실해짐.
