# 0007. 콘텐츠 합성 원칙: 외부 자료 + LLM + 본인 경험

- **Status**: Accepted
- **Date**: 2026-05-03

## Context

각 노드 콘텐츠를 어떻게 생산할 것인가. 후보:

1. 100% 자작
2. LLM 초안 + 본인 검수
3. 외부 자료 큐레이션 + 본인 해설
4. 세 가지 합성

## Decision

**세 가지 합성**, 각 소스에 명확한 역할 부여:

| 소스                                                           | 역할                            | 비중    |
| -------------------------------------------------------------- | ------------------------------- | ------- |
| 외부 자료 (MDN, web.dev, 공식 문서, 토스/카카오 기술블로그 등) | 1차 출처·정확성 보증            | 30%     |
| LLM                                                            | 구조 초안·예시 코드·표현 다듬기 | 30%     |
| 본인 실무 경험                                                 | "실무에서 왜 중요한가"의 차별화 | **40%** |

## Consequences

### 긍정

- 정확성: 외부 자료가 사실 보증
- 속도: LLM이 구조·예시 빠르게
- 차별화: 본인 경험이 어디에도 없는 가치 (가장 큰 비중)
- 검수 비용 합리적: 본인 경험은 검수 불필요, 외부 자료는 출처 확인만, LLM 부분만 깊은 검수

### 부정

- 세 소스 합성에 워크플로 필요 (CONTENT_SCHEMA.md에 명시)
- 본인 경험이 부족한 노드는 차별화 약함 → status: draft 유지

## 거부된 방향

### 100% 자작

- 노드 80~120개 자작은 1인 개발 비현실적
- 시간 보존을 위해 LLM·외부 자료 활용 필수

### LLM에 통째로 맡기기

- 그럴듯하지만 미묘하게 틀린 설명 빈번
- 검수 비용이 작성 비용 초과 가능
- 본인 경험 부분 결손 → 차별화 사라짐

### 외부 자료 요약만

- "MDN 번역 아닌가" 비판
- 저작권 이슈
- 본인 경험 부재로 차별화 약화

## 본인 경험 활용 가이드

다음 영역을 적극 활용:

- **OfficeNEXT B2B 메신저** 개발 경험 (Jiran Jigyo Software)
- **IndexedDB 캐싱** 구현
- **Web Worker** 활용 성능 최적화
- **Electron** 크로스 플랫폼 데스크톱
- **RN/Expo** 모바일 앱 (FE 외 영역이지만 비교 관점에서)
- **Next.js / Zustand / TypeScript** 실무 패턴

각 노드에서 "실무 포인트" 섹션은 가능한 한 구체 사례·수치·전후 비교 포함.

## 안티패턴

- ❌ 본인 경험 섹션을 LLM이 가짜로 생성하게 두기 (신뢰 무너짐)
- ❌ 외부 자료 문장 직역 (저작권 + 차별화)
- ❌ LLM 초안 그대로 publish (미묘한 오류)
- ❌ 본인 경험만 (일반화·체계화 부족)

세 소스의 *합성*이 핵심. 어느 한쪽이 결손되면 노드 publish 보류.
